# IOCP Server Engine: 프로젝트 트러블슈팅 일지 (Troubleshooting Log)

본 문서는 IOCP 게임 서버 개발 과정에서 발생한 버그의 원인을 추적하고 해결한 과정을 기록한 문서입니다.

---

## [Case 1] 타임존 연산 오버헤드로 인한 비동기 I/O 핸들 파괴 분석

### 1.1 문제 상황

* **현상:** 로그 출력 포맷을 개선한 후, `IOCP` 서버의 `WSARecv` 호출 단계에서 `ERROR_PATH_NOT_FOUND (ErrorCode 3)` 에러가 발생함.
* **영향:** 클라이언트의 접속(`Accept`)은 정상적으로 이루어지나, `GQCS` 이후 첫 데이터를 수신하기 직전에 소켓 통로가 파괴되며 연결이 강제 종료됨.

### 1.2 디버깅 과정

#### 1.2.1 고정관념에 의한 오판

* **초기 대응:** 발생한 에러 코드가 커널 레벨의 핸들 에러(`ErrorCode 3`)였기 때문에, 단순한 로그 출력 함수가 원인일 것이라고는 전혀 상상하지 못함.
* **시행착오 및 지체 원인:**
  * **소켓 API 수준의 검토:** `ERROR_PATH_NOT_FOUND`라는 에러 메시지에 매몰되어, 소켓 생성 시점부터 `AcceptEx` 호출 이후의 `SO_UPDATE_ACCEPT_CONTEXT` 옵션 적용까지 소켓 API의 호출 순서와 파라미터를 중점적으로 검토함.
  * **엔진 내부 로직 분석:** 소켓 호출부에 결함이 발견되지 않음에도 로그는 로직에 영향이 없을 것이라는 고정관념에 사로잡혀, 세션 관리나 핸들링 로직 등 엔진 내부의 복잡한 상태 관리 코드를 분석하며 상당한 시간을 소모함.

#### 1.2.2 가설의 전환과 `git bisect` 도입

* 현재 로직에 문제가 없다면, 이전 커밋 어딘가에 잠재되어 있던 설계 결함이 환경 변화(로그 수정 등)를 만나 터진 것이라 가설을 수정함.
* 잠재적 버그가 시작된 지점을 찾기 위해 과거 커밋들을 대상으로 `git bisect`를 실행하여 이분 탐색을 진행함.
* **결과:** `git bisect`가 지목한 First Bad Commit은 아래와 같음.
  * **Bad Commit Hash:** [`f35dfca`](https://github.com/sonho00/iocp-server-engine/commit/f35dfcaa3ded9264777df9c311a275f6a4c53424) (Refactor: 단계별 로그 시스템 구축 및 인터페이스 표준화)

#### 1.2.3 지점 특정

* 도구의 결과를 바탕으로 편견을 버리고 로그 코드를 분석함.
* `GQCS` 직후 호출되는 `LOG_DEBUG`를 주석 처리했을 때 에러가 사라지는 것을 확인하고 최종 원인을 확정함.

### 1.3 원인 분석

#### 1.3.1 `std::chrono::current_zone()`의 오버헤드

* `LOG_DEBUG` 내부에서 로컬 시간을 구하기 위해 호출한 `std::chrono::current_zone()`이 MSVC 환경에서 전역 락 경합을 유발함. 이로 인해 I/O를 처리해야 하는 워커 스레드들이 Hot Path에서 수 밀리초 이상 지연되었고, 서버네트워크 스택이 이를 감당하지 못해 소켓 연결을 강제로 파괴함. 이 시점에 커널 큐에는 1236 (`ERROR_CONNECTION_ABORTED`) 에러 패킷이 쌓임.

#### 1.3.2 에러 코드 가로채기

* 연결이 터졌으므로 다음 루프에서 `GQCS`는 당연히 실패를 반환함. 이때 원래대로라면 `WSAGetLastError()`를 통해 1236(`ERROR_CONNECTION_ABORTED`)을 얻어야 했으나, 에러 코드를 백업하기 전 윗라인에 위치한 `LOG_DEBUG`가 먼저 실행되는 코드 순서의 결함이 있었음.

#### 1.3.3 최종 오염

* `LOG_DEBUG`가 돌면서 내부 타임존 조회를 위해 존재하지 않는 시스템 경로를 Probing 하였고, 이 과정에서 스레드 전역 마커인 `GetLastError()`가 3 (`ERROR_PATH_NOT_FOUND`)으로 덮어써짐. 결국 뒤늦게 호출된 `WSAGetLastError()`가 오염된 에러코드 3번을 읽어 들여 디버깅에 혼선을 초래함.

### 1.4 해결 방법

* `WSAGetLastError()` 호출 단계 전, 전역 에러 마커를 오염시킬 수 있는 그 어떤 함수도 끼어들지 못하도록 에러 코드 백업 순서를 최우선으로 조정함.
* 무거운 `std::chrono::current_zone()` 호출을 제거하고, 오버헤드가 거의 없는 `std::chrono::system_clock` 기반의 가벼운 타임스탬프 로직으로 교체함.
* **추가 검증:** 동일한 코드를 부하가 적은 구간(`Main`)에서 실행했을 때는 문제가 없음을 확인. 이는 함수 자체가 틀린 것이 아니라 사용된 위치가 문제였음을 입증함.

### 1.5 교훈

#### 1.5.1 에러 코드 백업 타이밍의 즉각성 확보

* Win32/WSA API 환경에서 함수 호출 실패를 판정한 직후, 다른 함수를 실행하기 전에 가장 먼저 에러 코드를 변수에 안전하게 백업해야 함. 중간에 끼어든 무심한 표준 라이브러리 함수가 내부 탐색/예외 처리 도중 `GetLastError()` 마커를 오염시켜 진짜 일어난 문제의 핵심 원인을 은폐하고 착시를 유발할 수 있음.

#### 1.5.2 Hot Path에서의 오버헤드 경계

* 아무리 표준 라이브러리가 제공하는 검증된 기능이라 하더라도 동작 원리와 실행 비용을 명확히 파악하지 않은 상태로 `GQCS` 직후 등의 민감한 영역에서 남발해서는 안 됨. 타임존 파일 탐색처럼 내부 레지스트리 조회나 전역 락 경합이 발생하는 기능은 언제든 스레드를 멈춰 세우고 멀쩡하던 네트워크 소켓을 강제 차단시키는 원인이 될 수 있음.

## [Case 2] Sparse Set 락 누락으로 인한 세션 상태 불일치 분석

### 2.1 문제 상황

* **현상:** 재접속 스트레스 테스트 중 특정 상태의 파티션에 세션이 누적되며 세션 풀이 고갈되어 테스트가 중단됨.
* **영향:** 세션이 정상적으로 상태 전환되지 않아 신규 접속을 처리할 수 없는 상태가 됨.

### 2.2 원인 분석

#### 2.2.1 락 누락의 배경

* 세션 상태를 `idle`, `pending`, `connected`, `disconnecting` 네 파티션으로 관리하는 확장된 Sparse Set 구조를 사용.
* Sparse 배열은 핸들을 인덱스로 Dense 배열의 위치를 조회하는 용도로, 인덱스가 정해지면 해당 슬롯은 그 핸들만 사용한다고 판단하여 락이 불필요하다고 판단함.

#### 2.2.2 실제 문제

* Sparse 배열과 Dense 배열은 항상 쌍으로 갱신되어야 일관성이 유지되는 구조.
* Dense에만 락을 적용했기 때문에 두 배열의 갱신이 원자적으로 이루어지지 않아, 중간 상태에서 다른 스레드가 접근하면 상태 전환이 틀어짐.

### 2.3 해결 방법

* Sparse 배열 접근에도 Dense와 동일한 락을 적용하여 두 배열의 갱신이 원자적으로 이루어지도록 수정.

### 2.4 교훈

* "이 슬롯은 나만 쓴다"는 가정은 Sparse 배열 단독으로는 성립할 수 있음.
  하지만 Sparse와 Dense가 항상 함께 갱신되어야 하는 구조에서는
  두 배열 전체를 하나의 임계 영역으로 묶어야 함.

## [Case 3] 서버 종료 시 객체 수명 주기 및 멤버 변수 소멸 순서로 인한 크래시 분석

### 3.1 문제 상황

* **현상:** 서버 종료 시 매직 버퍼(`MagicBuffer`)의 `Clear` 함수에서 크래시가 발생함.
* **영향:** 프로세스가 안전하게 리소스를 정리하고 정상 종료되지 못하며 메모리 오염 흔적을 남기고 비정상 종료됨.

### 3.2 디버깅 과정

#### 3.2.1 초기 가정의 오류 (IOCP Abort 작동에 대한 착각)

* `IocpCore`가 소멸하며 서버가 내려갈 때, 대기 중이던 모든 비동기 I/O가 `OPERATION_ABORTED`로 안전하게 리턴되고 워커 스레드가 이를 차례로 정리할 것이라 가정함.
* `SessionManager`가 들고 있던 세션 포인터 배열(`sessionPtrs_`)이 소멸하는 타이밍에 맞추어 세션의 스마트 포인터 참조 카운트가 완벽하게 0이 되어 자원들이 자동으로 청소될 것이라 안일하게 판단함.

#### 3.2.2 실제 원인 추적

* 워커 스레드가 정상 종료 시 완료 패킷(`Abort` 이벤트들)을 큐에서 처리하지 못한 채 스레드 루프를 탈출해 조기 종료되었고, `OverlappedEx`들이 스마트 포인터 참조를 여전히 쥐고 있었음.
* 이 상태에서 세션 매니저의 소멸 주기와 자원 풀(`ObjectPool`)의 소멸 루틴이 꼬이면서 세션의 강제 소멸자가 작동함.
* 세션이 강제로 소멸하는 도중에 내부 멤버 변수들 간의 소멸 순서 및 참조 카운트 해제 타이밍의 부작용으로 이미 파괴된 영역을 또 한 번 건드려 크래시가 유발됨을 확인함.

### 3.3 원인 분석

#### 3.3.1 객체 및 컨테이너 소멸 순서 역전

* C++ 규칙에 의해 클래스 내부 멤버 변수는 선언된 순서의 역순으로 소멸함.
* `SessionManager` 클래스에서 `sessionPool_`이 `sessionPtrs_` 배열보다 윗라인에 선언되어 있어, `sessionPtrs_` 배열이 먼저 파괴되어 매니저 측 포인터들을 해제함.
* 하지만 워커 스레드가 정상적으로 해제하지 못한 잔여 스마트 포인터 참조가 살아있었기 때문에 세션 자체는 완벽하게 지워지지 않고 메모리에 대기함.

#### 3.3.2 풀 소멸자에 의한 강제 소멸자 호출

* 뒤이어 위에 있던 `sessionPool_`의 소멸자가 호출됨. `ObjectPool.hpp` 내부의 소멸자는 메모리에 생성되어 있던 세션들의 소멸자를 명시적으로 직접 강제 호출함.

#### 3.3.3 소멸자 실행 도중 참조 카운트 0 도달

* `Session` 클래스 내부에서 `Read Overlapped(ReadOV)` 변수가 `Write Overlapped(WriteOV)` 변수보다 위에 선언되어 있었음.
* 세션 소멸자가 실행되면서 아래에 있던 `WriteOV`가 먼저 파괴되어 무효한 메모리가 됨.
* 뒤이어 위에 있던 `ReadOV`가 파괴될 차례가 됨. `ReadOV`가 파괴되면서 마지막으로 자신이 보관하고 있던 `SharedPoolPtr<Session>`을 해제함.
* 이 순간 해당 세션의 참조 카운트가 0이 되는 트리거가 소멸자 동작 중간에 발동하면서, 스마트 포인터의 커스텀 딜리터가 호출되어 이미 파괴가 완료된 `WriteOV`를 리셋하려 하여 `WriteOV`의 버퍼를  건드림. 결국 이미 죽은 쓰레기 값을 참조(`Use-After-Free`)하게 되어 크래시가 발생함.

### 3.4 해결 방법

* 소멸자 실행 도중 참조 카운트 0이 되더라도 객체를 초기화하면서 이미 해제된 멤버 변수를 건드리지 않도록 변경.
* 풀에 객체를 반납할 때 초기화 하던 구조를 객체를 대여할때 초기화하는 구조로 변경하여, 소멸자에서 객체를 초기화하는 구조를 완전히 제거함.

### 3.5 교훈

#### 3.5.1 자원 초기화 책임을 대여 시점으로 분리
* 소멸자 내부에서 객체를 재사용하겠다고 멤버 변수나 스마트 포인터를 정리하다가 참조 카운트가 0이 되면, 스마트 포인터가 자원 정리 로직을 이중으로 건드려 크래시가 날 수 있음.
* 소멸 시점에는 최소한의 해제만 하고, 객체를 풀에서 다시 대여할 때 초기화를 하도록 책임을 확실히 나누는 게 안전함.

#### 3.5.2 서버 종료 시 스레드 종료와 리소스 수명의 분리
* 서버 종료 시 워커 스레드가 남은 비동기 작업을 전부 다 처리하고 끝나는 것이 아님.
* 플래그 등으로 인해 스레드가 먼저 루프를 탈출해서 종료되더라도, 미처 처리되지 못한 `OverlappedEx`가 참조를 쥐고 있다면 리소스의 수명은 스레드 종료 이후까지 남아있게 됨.
* 스레드가 끝나는 시점과 리소스가 살아있는 시점은 완전히 별개라는 것을 인지하고 설계해야 함.
